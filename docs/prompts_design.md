# 🎨 プロンプト設計の意図

このドキュメントは `prompts/` 配下にある 4 種類のプロンプトの**設計理由**と**改変ガイド**をまとめたものです。

**対象読者**:
- プロンプトを自分好みに調整したいユーザー (中級以上)
- 別ドメインに転用するときのリファレンス
- AI コーディングエージェント (将来の改良セッション)

---

## 1. プロンプト一覧

| ファイル | 役割 | タイミング |
|---|---|---|
| `relevance_check.md` | 関連度スコアリング (0-10) | バッチ処理、記事取得直後 |
| `digest_hybrid.md` | ダイジェスト生成 (ハイブリッドモード、デフォルト) | スコアリング後、`mode: hybrid` のとき |
| `digest_beginner.md` | ダイジェスト生成 (初心者モード) | `mode: beginner` のとき |
| `digest_expert.md` | ダイジェスト生成 (上級者モード) | `mode: expert` のとき |

---

## 2. 共通原則

4 つのプロンプト全てに適用している設計原則:

### 原則 1: テンプレート変数で config を注入

プロンプト内の `{{HOLDINGS}}`, `{{INTERESTS}}`, `{{EXCLUDES}}`, `{{ARTICLES_JSON}}`, `{{DELIVERY_DATETIME}}` は
`scripts/summarize.py::_render` で実行時に差し替えられます。

**利点**:
- ユーザーの保有銘柄情報を AI の「システムプロンプト」相当に組み込める
- プロンプト本体は編集しやすい Markdown のまま保てる

### 原則 2: 口調のコントロール (「〜なのだ」禁止)

全てのダイジェストプロンプトに以下の指示を明記:

> 「〜なのだ」口調は**使わない**

**理由**: ずんだもんの三刀流マネーラボの**視聴者向け**ツールであり、配布先の視聴者プロフィール (20-40 代会社員) にキャラクター口調は親和しない。主本人の動画内ではキャラ口調を使うが、個人用のデジタル秘書としては**中立で読みやすい敬体/常体**が最適。

### 原則 3: JSON 入出力のロバスト性

スコアリングプロンプトは JSON で返させるが、Gemini / Claude どちらもたまに前後に説明文を付けてしまう。

対策:
- プロンプト側で「**前後に説明文やコードフェンスを付けないでください**」と明示
- パース側 (`_extract_json`) でもコードフェンス剥がし + `{` 〜 `}` 抽出のフォールバックを実装

### 原則 4: トークン経済性

- スコアリング時: 記事本文は 600 字、サマリ 300 字までに制限 → 1 バッチ (20 記事) で入力 10-15K トークン程度
- ダイジェスト生成時: top/notable に絞った後の 3〜10 記事のみを本文 1500 字で渡す → 入力 5-8K トークン程度

Gemini Flash の 1 分 15 req 無料枠 & 1 日 100 万トークン枠に十分収まる設計。

## 3. `relevance_check.md`

### 目的

取得した全記事に **0〜10 点** のスコアを付けて、**top (必読)** / **notable (その他注目)** / **除外** に仕分ける。

### 入力

```json
{
  "HOLDINGS": "NVDA (エヌビディア), AAPL (アップル), ...",
  "INTERESTS": "S&P500, 新NISA, 半導体, ...",
  "EXCLUDES": "仮想通貨, IPO",
  "ARTICLES_JSON": [
    {"id": 0, "title": "...", "source": "...", "body_excerpt": "..."},
    ...
  ]
}
```

### 出力

```json
{
  "results": [
    {"id": 0, "score": 9, "reason": "保有NVDAの決算発表"},
    {"id": 1, "score": 3, "reason": "一般ニュース"}
  ]
}
```

### スコア基準の設計意図

| 点数 | 記事の例 |
|---|---|
| 10 | 保有銘柄の決算 / ガイダンス修正 / M&A / 規制 |
| 8〜9 | 保有セクター全体の大きな動き (半導体全般等) |
| 6〜7 | 興味分野のマクロニュース (FRB 利下げ等) |
| 4〜5 | 一般市場ニュース |
| 1〜3 | ほぼ無関係 |
| 0 | 除外トピック / 広告 / 重複 |

この連続的スケールにした理由:
- `score_threshold_top=7` / `score_threshold_notable=5` を **config.yaml で可変**にするため
- 閾値判定を AI 側でなく後工程 (Python) でやることで、ユーザーが挙動を予測・調整しやすい

### 改変のヒント

- **スコアが辛すぎる / 甘すぎる**: プロンプト中の「7 点以上 = 必読」の説明文を調整 ("ほぼ直接的" → "完全に直接的" 等)
- **理由を長く**: `reason` の「20 字以内」制約を外す (ただしトークン増)
- **英語記事にも対応**: 入力 articles に `language` フィールドを足してプロンプトに含める

## 4. `digest_hybrid.md` (デフォルト)

### 目的

初心者にも上級者にも**バランスよく読める**ダイジェストを生成。

### 設計

- **top セクション**: 見出し + 短い初心者向け解釈 + 出典URL
- **notable セクション**: 見出しのみを列挙、URL は省略 (一覧性)

### 文字数制限

- 全体 1500 字以内 (Discord 1 embed の 4000 字制約に余裕を持たせる)

### なぜハイブリッドがデフォルトか

- 初心者: 専門用語に出会ったら躓く → `💡 初心者向け` で解釈を 1 行ヘルプ
- 上級者: 必要十分な情報が見出し + URL で取れる
- **両立コスト**: 追加 100〜200 字/記事のみ。初心者モードよりはるかに短い

### 改変のヒント

- **top の記事数を 5 個に増やしたい**: `config.yaml.limits.top_count: 5`
- **絵文字を変えたい**: プロンプト内の 📰 🔥 📊 💡 📎 └ を好みのものに差し替え
- **URL を notable にも付けたい**: プロンプトの notable フォーマット説明に `📎 <URL>` を追加

## 5. `digest_beginner.md`

### 目的

投資初心者が**不安にならず読める**丁寧な解説形式。

### 4 ブロック構造

```
【何が起きた?】
【どういう意味?】
【あなたへの影響】
【専門用語解説】
```

### 設計判断

- **4 つ決まった見出し** → AI が出力ブレを起こしにくい
- **「あなたへの影響」** → 自分の保有銘柄との紐付けを強制的に書かせる
- **「買うべき/売るべき」禁止** → 投資助言にならないよう予防線 (規制リスク回避も兼ねる)
- **煽り調禁止** → 「暴落!」「衝撃!」のような感情ワードを抑止

### 文字数制限

- 全体 2000 字 (情報量優先)

### 改変のヒント

- **ブロックを減らす**: `【専門用語解説】` 部分を削除
- **ブロックを足す**: `【今日のアクション (推奨なし、情報のみ)】` 等
- **口調を変える**: 「です・ます」 → 「だ・である」

## 6. `digest_expert.md`

### 目的

上級者向けに**数字とファクトだけ**を簡潔に。

### 特徴

- `Revenue / EPS / ガイダンス / マクロ指標` の数値抽出を AI に指示
- **推測・感情ワード禁止**
- 全体 1200 字 (最も短い)

### 改変のヒント

- **さらに短く**: 「全体 800 字」に書き換え、notable は見出しのみ
- **英語表記にしたい**: 「ティッカーは英字のまま」「数値は $ 表記優先」等を追加

## 7. 共通の注意点: 出力品質のデバッグ

### AI の出力が期待通りにならないときの診断手順

1. `scripts/run.py --dry-run --save-raw raw.json` でスコアと記事をダンプ
2. `raw.json` を開いて AI のスコアが妥当か見る
3. スコアが妥当なら → ダイジェストプロンプトの問題
4. スコアがおかしい → `prompts/relevance_check.md` の基準を見直す

### プロンプトのイテレーション

編集 → commit → `workflow_dispatch` で手動実行 → Discord で確認 のサイクル。
GitHub UI 上で完結するので 3 分/イテレーション。

---

## 8. 新しいモードを足したいとき

例: `mode: advisor` (投資助言風、自己責任前提)

1. `prompts/digest_advisor.md` を作成
2. `scripts/summarize.py` の `Mode` Literal に `"advisor"` を追加
3. `config.yaml` で `mode: advisor` に設定
4. コミット

`build_digest` は `f"digest_{mode}"` でファイル名を解決するので、プロンプトファイルを置くだけで機能追加できる設計にしてあります。
